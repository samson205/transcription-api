# Transcription API

REST API для асинхронной транскрипции разговоров оператора с клиентом с использованием Faster-Whisper и Pyannote Audio.

Проект предназначен для:

- транскрипции аудиозаписей;
- диаризации (разделения речи по спикерам);
- идентификации операторов по голосовым эмбеддингам.

## Стек технологий

- Python 3.12
- FastAPI
- PostgreSQL + pgvector
- Redis
- Celery
- Faster-Whisper
- Pyannote Audio
- SpeechBrain
- Docker
- Docker Compose

## Архитектура

Проект построен с разделением ответственности между слоями приложения.

```
api/
├── core/             # Конфигурация приложения
├── engines/          # ML-модели
├── models/           # SQLAlchemy модели
├── orchestrators/    # Оркестрация бизнес-процессов
├── processors/       # Обработка результатов моделей
├── repositories/     # Работа с БД
├── routers/          # REST API
├── schemas/          # Pydantic схемы
├── services/         # Бизнес-логика
└── tasks/            # Celery задачи
```

## Hugging Face (upd: 15.07.2026)

Модель `pyannote/wespeaker-voxceleb-resnet34-LM` скачивается автоматически
при первом запуске, токен для неё не требуется — она в открытом доступе.

Опционально можно указать токен Hugging Face в `.env` (`HF_TOKEN`) — это
снимает более строгие лимиты на количество запросов к Hugging Face Hub для
анонимных пользователей и пригодится, если в будущем проект вернётся к
моделям с ограниченным доступом.

## Запуск

### 1. Клонировать репозиторий

```bash
git clone https://github.com/samson205/transcription-api
cd transcription-api
```

### 2. Заполнить файл `.env`

Пример:

```env
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres
POSTGRES_DB=transcription

HF_TOKEN=your_huggingface_token
...
```

При необходимости заполните остальные параметры конфигурации.

### 3. Собрать проект

```bash
docker compose build
```

Первая сборка может занимать продолжительное время.

> **Важно**
>
> Проект использует PyTorch, Faster-Whisper, Pyannote Audio и CUDA-библиотеки. Во время первой сборки Docker может потребоваться **до 30 ГБ свободного места** для хранения промежуточных слоев и зависимостей. После завершения сборки неиспользуемые слои можно удалить командой:
>
> ```bash
> docker builder prune
> ```

### 4. Запустить контейнеры

```bash
docker compose up
```

Будут запущены:

- API
- Celery Worker
- PostgreSQL
- Redis

После успешного запуска документация OpenAPI будет доступна по адресу:

**http://localhost:8000/docs**

## Используемые модели

Проект автоматически загружает необходимые модели при первом запуске.

Используются:

- Faster-Whisper
- Pyannote Audio
- SpeechBrain ECAPA

Модели сохраняются локально и повторно не скачиваются.

## Возможности

- Транскрипция аудио

- Очередь задач через Celery

## CLI-утилита

Помимо REST API, в проекте есть консольная утилита `api/cli.py` для пакетной
обработки уже имеющихся исторических записей звонков — без необходимости
поднимать очередь задач через HTTP-запросы по одному файлу.

> **Важно:** CLI не входит в `docker-compose.yaml` и не запускается как
> отдельный контейнер. Она рассчитана на локальный запуск через `venv`,
> **на Linux** (или WSL2 на Windows — нативный Windows не поддерживается
> из-за проблем совместимости CUDA/torch-сборок для CLI-сценария).
> Сервис `db` при этом всё равно нужен — он поднимается через
> Docker, а CLI обращается к нему извне контейнеров.

### Возможности

- обработка одного аудиофайла с сохранением результата в JSON;
- пакетная обработка всех аудиофайлов в директории;
- регистрация оператора и его голосового эмбеддинга по аудиозаписи.

### Настройка окружения

1. Поднять только БД (без API, Redis и воркера — их CLI не использует):

   ```bash
   docker compose up -d db
   ```

2. Создать и активировать виртуальное окружение:

   ```bash
   python3.12 -m venv .venv
   source .venv/bin/activate
   ```

3. Установить зависимости:

   ```bash
   pip install -r requirements.txt
   ```

   > Первая установка может занять продолжительное время и много места на
   > диске — по тем же причинам, что и сборка Docker-образа (torch,
   > faster-whisper, pyannote и CUDA-зависимости).

4. Указать в `.env` адрес БД **как для localhost**, а не как для
   Docker-сети (значение отличается от того, что использует сам
   docker-compose для контейнеров `api`/`worker`):

   ```env
   DB_HOST=localhost
   ```

5. Модели, уже скачанные через Docker-запуск, переиспользуются автоматически
   — каталог `ml_models/` примонтирован в контейнер `api` как обычная папка
   в корне проекта, так что при локальном запуске CLI найдёт их там же, без
   повторной загрузки.

### Примеры использования

Обработка одного файла с сохранением результата в JSON:

```bash
python -m api.cli -f /path/to/call.mp3 -o ./output
```

Пакетная обработка всех записей в директории:

```bash
python -m api.cli -d /path/to/calls_archive -o ./output
```

Регистрация оператора по голосовому образцу:

```bash
python -m api.cli -f /path/to/operator_sample.mp3 --operator-name "Иван Иванов"
```

Полный список флагов:

```bash
python -m api.cli --help
```

### Ограничения

- поддерживаются только расширения из `EXTENSIONS` в конфиге (`.mp3`,
  `.wav`, `.ogg`);
- `--operator-name` работает только вместе с `-f`/`--file` — при пакетной
  обработке директории (`-d`) флаг недопустим;
- обработка идёт последовательно, файл за файлом — для больших архивов
  это может занять значительное время в зависимости от доступного
  GPU/CPU.
